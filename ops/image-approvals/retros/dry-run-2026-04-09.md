## The single most important finding

**The V1 homepage's existing Pulpit Rock image is a drone shot.** EXIF on the Commons file records camera manufacturer DJI, model FC3582 (a Mavic 3 body), focal length 6.72mm, ISO 130, 1/3200s — definitively an aerial drone photograph. The `home-cover-01` brief explicitly hard-rejects drone imagery.

This means the one image the V1 site has been using as its Peninsula landscape signature **would not survive the V2 approval workflow as currently specified**. This is exactly the kind of thing the dry run existed to surface.

The editor has three paths:

1. **Keep the drone reject.** V2 needs a different Pulpit Rock / Cape Schanck image. The SA-licensed Marshlung "looking towards Pulpit Rock at dawn" is the best available alternative and is carried as `home-cover-01-a`.
2. **Relax the drone reject for this one asset.** This would restore access to the cleanest-licence Peninsula landscape on Commons, but breaks a brief-level rule on the site's single most important surface.
3. **Defer home-cover-01 entirely.** Keep the fog placeholder on the homepage until Visit Victoria Content Hub delivers a verified, ground-level cover image in weeks 2–3.

**Sourcing-agent recommendation:** path 1 (keep the drone reject, approve candidate A with SA override, move on). The SA override is a smaller editorial compromise than publishing a drone shot on the brand's signature image, and the share-alike obligation on a single image file is operationally manageable.

## Format / workflow issues surfaced by the dry run

### 1. The `shortlist-template.json` schema has no field for "blocked with reason"

The template assumes every shortlist has ≥1 candidate. I added a `status` field + `sourcingNotes.blocker` + `sourcingNotes.recommendedUnblockPath` fields to the four blocker shortlists to keep them machine-parseable, but the template should be formally updated to make this first-class. Suggested change:

```diff
  "slotId": "{slot-id}",
  "createdAt": "{ISO timestamp}",
  "createdBy": "{agent-id}",
  "briefPath": "ops/image-approvals/briefs/{slot-id}.md",
+ "status": "shortlisted | blocked-no-candidates | deferred",
+ "sourcingNotes": {
+   "poolAssessment": "string",
+   "blocker": "string (only if status == blocked)",
+   "recommendedUnblockPath": "string or array"
+ },
  "candidates": [ ... ]
```

### 2. The `index.csv` state model needs a `blocked` state

Current state transition table (workflow doc §README transitions):

> `briefed` → `shortlisted` → `approved` / `rejected-all` → `implemented`

Reality exposed a fourth route: `briefed` → `blocked-no-candidates`. I used that string in the index.csv. Suggested addition to the transitions table:

| From | To | Who | Trigger |
|---|---|---|---|
| `briefed` | `blocked-no-candidates` | Sourcing agent | Writes `shortlists/{slot-id}.json` with `status: blocked-no-candidates` |
| `blocked-no-candidates` | `briefed` | Editor | After resolving the blocker (e.g. unblocks a source, authorises generative batch) |

### 3. Twinning risk is not enforced by the workflow

Candidate A of `home-cover-01` is the same underlying Commons image as candidate A of `place-cape-schanck-01`, and candidate A of `wine-hub-hero-01` is the same image as candidate A of `place-red-hill-01`. The workflow does not currently prevent the same image being approved to two slots by two different decisions. For a ~60-image first pass with a thin clean-licence pool, this will recur.

Suggested fix: on the editor review script (workflow §12, tooling stub 3), warn if a candidate's `directUrl` is already present in another slot's shortlist. Simple lookup, zero false positives.

### 4. The Wikimedia Commons clean-licence pool is thinner than sourcing report §3.3 implied

The sourcing report expected a ~10–15% hit rate on Commons. The real ratio for Peninsula-verifiable editorial-grade LANDSCAPE imagery is closer to 5% clean-licence (CC-BY / CC0 / PD) and 20% CC-BY-SA. The SA share-alike contamination question is therefore a gating decision for week 1, not an edge case. The sourcing report should be updated to call this out.
