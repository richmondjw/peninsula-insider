2. Strategic Operating Principles

These govern how we build. Treat them as non-negotiable.

  1. Atomic or nothing. Never add a venue, experience, walk, beach or hotel to a card-grid page again. From Week 1 onward, every new entity ships as its own URL with its own JSON record. The directory pages become generated views of the database, not the database itself.

  2. Photography commissions before site features. A new product surface (map, "This Weekend," Stay expansion) does not ship until at least 60% of its featured venues have original photography. We refuse to build premium UI on stock imagery — it betrays the brand promise faster than anything else.

  3. Ship every Thursday. Publishing cadence is the discipline that creates everything else. Every Thursday at 7am AEST: a new Journal piece, a newsletter issue, and a deploy. The cadence is more important than any single shipped feature. Miss a Thursday and the brand promise erodes.

  4. The newsletter is product, not afterthought. "The Insider's Letter" is treated as a first-class product line with its own KPIs, not as a footer form. Every editorial decision asks: does this strengthen the newsletter?

  5. Every recommendation is bookable. From Phase 2 onward, no venue page ships without an outbound booking action. If the venue has no booking system, we link to phone/website with affiliate parameters where available. A recommendation without a path to action is editorial waste.

  6. Decisions over options. James's job is to be the editor and the dealmaker, not to weigh tech tradeoffs in Slack threads. Every section of this roadmap recommends one answer and defends it. Reversals happen if and only if the data demands them.

  7. AI-agent leverage is the production model. The architecture must let Remy and specialist subagents (research, design, editorial, photo briefing) author venue records, draft articles, generate intent pages, and propose itineraries directly into the content layer with minimal human handoff. If a tool requires a human GUI to author content, it's the wrong tool.

4. Phase 1 — Atomic Foundations (Weeks 1–4)

This is the most important phase. Get this right and everything in Phases 2–4 becomes mechanical execution. Get it wrong and we ship a faster version of the same problem.

4.1 Atomic page architecture — the recommendation

Recommendation: Astro + JSON content collections + Sanity (lite) for editorial chrome only.

The full stack:

  • Astro as the static site generator. All listing pages, venue pages, intent pages, place pages, itinerary pages and journal articles are generated at build time from JSON files in the repo.
  • Content lives as JSON in src/content/ collections (venues/, experiences/, places/, itineraries/, articles/, events/, authors/). Astro Content Collections give us typed schemas, validation, and zero-runtime queries.
  • Sanity Studio (free tier) mounted at /admin purely for the Journal articles where James wants WYSIWYG editing — articles publish as JSON exports back into the repo via a build hook. Venues and structured entities are NOT edited in Sanity; they're authored by AI agents into JSON files directly.

Three-bullet rationale:

  • AI agents author into JSON natively. This is the biggest win. A research subagent can produce a complete venue JSON record in 30 seconds, validated against a Zod schema, committed to git. No CMS GUI to navigate, no API rate limits, no human in the loop for 200 venue records. This single decision is what makes solo-founder + AI velocity possible.
  • Astro is editorially confident and fast. Zero-JS by default for content pages, islands for the map and search. The output ships static to any CDN. Build times for 500+ pages stay under 90 seconds. The community templates lean editorial (vs. Next.js which leans app).
  • Hybrid lets us escape commitment. JSON in git is portable. If we outgrow this in 18 months and need a real CMS, we export to Sanity/Payload in a weekend. We are not locked in. Compare this to picking a heavyweight CMS up front and discovering its editorial workflow doesn't fit a media brand.

What we explicitly reject: Full Sanity-as-source-of-truth (too much GUI friction for AI authorship); Next.js (overkill — we don't need an app, we need a publication); Hugo/11ty (less component composition than Astro for the editorial design system); a pure database approach with Postgres + Prisma (operational overhead with no upside at this scale).

4.2 Content schema definition

Every entity is typed via Astro Content Collections + Zod. The schemas:

// src/content/config.ts (illustrative)
const venue = z.object({
  slug: z.string(),
  name: z.string(),
  type: z.enum(['restaurant','winery','cafe','bakery','pub','brewery','distillery','producer','market','hotel','villa','cottage','glamping','farm-stay','spa']),
  place: z.string(), // references places collection
  zone: z.enum(['bayside','hinterland','red-hill-plateau','ocean-coast','back-beaches','tip']),
  coordinates: z.object({ lat: z.number(), lng: z.number() }),
  address: z.string(),
  phone: z.string().optional(),
  website: z.string().url().optional(),
  bookingUrl: z.string().url().optional(),
  bookingProvider: z.enum(['opentable','tock','resy','sevenrooms','direct','booking.com','stayz','airbnb','peninsula-hot-springs','none']).optional(),
  priceBand: z.enum(['$','$$','$$$','$$$$']),
  hatRating: z.number().min(0).max(3).optional(),
  hallidayScore: z.number().optional(),
  awards: z.array(z.string()).optional(),
  signature: z.string(), // signature dish, wine, room
  editorNote: z.string(), // 2–3 paragraph opinion
  tags: z.object({
    mood: z.array(z.string()),    // long-lunch, anniversary, family, etc.
    season: z.array(z.string()),  // autumn, summer, all-year
    audience: z.array(z.string()) // couples, families, solo, group
  }),
  heroImage: z.string(),
  gallery: z.array(z.object({ src: z.string(), credit: z.string(), caption: z.string().optional() })),
  lastVerified: z.date(),
  publishedAt: z.date(),
  affiliateNote: z.string().optional() // disclosure
});

Plus parallel schemas for experience, place, article, itinerary, event, author. All defined in one file. All validated at build time. Schema breaking changes fail the build, so content cannot rot.

4.5 URL structure decision

Decision: place-included, category-prefixed URLs.

/eat-drink/red-hill/tedesca-osteria
/eat-drink/wineries/red-hill/montalto
/stay/merricks-north/jackalope
/do/wellness/peninsula-hot-springs
/do/walks/cape-schanck/bushrangers-bay

Three-bullet rationale:

  • Place is the most useful natural-language fragment for SEO. "Red Hill restaurants" is searched 20x more than "Mornington Peninsula restaurants in Red Hill." Putting place in the URL captures that.
  • Broadsheet uses this pattern and out-ranks everyone. No reason to invent a worse one. The strategic review explicitly flags this.
  • Refactor cost is high if wrong. This decision needs to be locked Day 1 of Week 1 because every venue JSON file's slug becomes a redirect debt the moment it ships.

The collection routing in Astro is one file:

// src/pages/eat-drink/[place]/[venue].astro

Categories that don't fit a single place (e.g. "walks" that span multiple zones) use the closest place anchor and a relatedPlaces[] field for cross-linking.

5.2 Newsletter launch — "The Insider's Letter"

Platform decision: Beehiiv.

Three-bullet rationale:

  • Beehiiv was built for newsletter-as-product brands (it powers Morning Brew, Milk Road and dozens of premium travel/food newsletters). Substack is a closed network with brand dilution; ConvertKit is creator-tooling without growth surfaces; MailerLite is functional but generic. Beehiiv has the boost + referral + monetization stack we need for Phase 4 paid tier.
  • Free up to 2,500 subs, $39/mo to ~10k, with built-in referral loops, paid subscription primitive, and a usable web archive. Cheapest credible path to a 5,000-subscriber list.
  • Beehiiv has a public API, so AI-agent-authored newsletter drafts can be pushed programmatically. Substack does not.

Positioning: "The Insider's Letter — every Thursday. The weekend ahead, what to book, what we're drinking. Five minutes. No fluff."

5.3 Map experience

Provider decision: Mapbox GL JS with a custom style.

Three-bullet rationale:

  • Mapbox custom styles are the only way to achieve the warm, editorial map aesthetic the brand needs. MapTiler is cheaper but its style editor is weaker and the basemaps look generic. Leaflet+OSM is free but shipping a credible custom-styled map on it is 4x the design effort.
  • 50,000 free monthly map loads on Mapbox is plenty for our traffic in Phase 2. Cost trip-wires only matter if we hit ~250k monthly visitors, which is a Phase 5+ problem.
  • Mapbox vector tiles + GL JS gives us smooth interaction on mobile. The geography of the Peninsula deserves proper rendering.

Custom style direction: Warm cream base (#f6f1e8), terracotta roads (#c45838), basalt-hued coastline (#3a4a52), muted forest greens (#7e8e6f) for parkland. The brand palette transferred to a map. ~1 day of design work in Mapbox Studio.

Integration approach:

  • Full-screen map at /map — every venue pinned, filtered by type/place/mood
  • Embedded 300px-tall map on every venue page (showing venue + 3 nearby)
  • Embedded route map on every itinerary page (numbered stops with drive time)
  • Map strip preview on the homepage linking to /map
  • Place hub pages (/eat-drink/red-hill/) show a filtered map for that place

Pin design: Custom SVG markers per venue type (knife & fork for restaurants, wine glass for wineries, key for stays, footprint for walks). Cluster at low zoom; expand at zoom 13+.

5.5 Bylines + named editor strategy

9. Tech Stack Decisions

The full slate. Each pick defended in three bullets. No options menus.

Static site generator / framework: Astro

  • Editorial-first by default — zero-JS content pages, islands for interactive bits (map, search, wizard), excellent build times
  • Content Collections + Zod schemas make AI-agent authorship trivial and validated
  • Component model is more flexible than 11ty/Hugo without the runtime overhead of Next.js — exactly the sweet spot for a publication

CMS approach: JSON in git, with Sanity Studio (lite) for Journal articles only

  • AI agents author venue/experience/place records directly into JSON — no GUI, no API rate limits, no human handoff
  • Sanity Studio mounted at /admin for the 1–2 articles/week that James wants to compose visually (his preference for editorial flow)
  • Zero lock-in — if we outgrow this, exporting to a heavyweight CMS is a weekend job

Map provider: Mapbox GL JS with custom brand style

  • Custom styling is the only way to land the brand-correct map aesthetic — generic OSM pins betray the brand
  • 50,000 free monthly map loads is generous for our traffic until we cross 200k monthly visitors
  • Vector tiles + smooth interaction on mobile — the geography of the Peninsula deserves proper rendering

Newsletter platform: Beehiiv

  • Built for newsletter-as-product brands (Morning Brew, Milk Road) — exactly the model we're following
  • Public API lets AI agents draft + push newsletter issues programmatically
  • Built-in paid subscription primitive — Phase 4 membership doesn't need a second platform

Booking integration platforms (prioritized): OpenTable → Booking.com → Stayz → Peninsula Hot Springs (direct) → Tock → GetYourGuide

  • OpenTable + Booking.com cover the highest commission volume with the lowest integration cost — start there
  • Direct deal with Peninsula Hot Springs is the single highest-leverage bespoke integration on the Peninsula
  • Tock + GetYourGuide cover the upmarket venue gap that OpenTable misses

Image hosting/CDN: Cloudinary

  • Free tier covers 25GB of bandwidth/month — comfortable for Phase 1–2
  • Built-in transforms (responsive sizing, WebP/AVIF, blur placeholders, focal-point cropping) — eliminates the entire image-pipeline build problem
  • DAM-style organization lets the photographer upload directly + Pixel agent retrieve by tag

Analytics: Plausible

  • Privacy-respecting + cookieless — appropriate for an editorial brand and avoids GDPR/cookie banner ugliness
  • Single-pane dashboard with real-time + historical — simpler than GA4
  • Cheap — $9/mo at our traffic, no upgrade pressure

Search: Pagefind (build it; do not buy)

  • Static search index built at compile time — zero infra, no API costs, ships with the site
  • Excellent performance for content sites under 10k entities
  • Algolia would cost $50–$500/mo for capabilities we don't need until Phase 5+

Hosting: Vercel (migrate from GitHub Pages)

  • GitHub Pages is fine for static HTML; it's wrong for Astro + image optimization + ISR + redirect rules
  • Vercel's preview deployments per branch are the killer feature for the parallel-build cutover plan
  • Free tier covers our traffic until Phase 5+; the migration is a one-day effort and gives us instant velocity
