# Peninsula Insider — Editor’s Workbench Phase 1 Implementation Brief

**Prepared for:** James Richmond  
**Prepared by:** Remy  
**Date:** 17 April 2026  
**Status:** Approved Phase 1 build brief  
**Purpose:** Define the first build scope for the Peninsula Insider Editor’s Workbench so it can move from strategy into implementation.

---

## 1. Build recommendation

Start with a **Phase 1 MVP**.

Do not try to build the whole Workbench in one go.
The first version should do one thing well:

**make Peninsula Insider’s live operating state visible in one place.**

That means the first build is a **truth surface**, not a full control plane yet.

---

## 2. Phase 1 objective

The first build should answer these questions immediately:
- What needs attention today?
- What is stale or broken?
- What is ready to publish?
- What is blocked?
- What do the major content hubs look like right now?
- What did the system change recently?

If it does that, it will already be highly valuable.

---

## 3. Recommended MVP modules

## Module 1 — Today’s Priorities
A top-level card or column showing:
- recommended fixes today
- recommended publishes today
- urgent issues
- held approvals

This should be the first thing James sees.

---

## Module 2 — Live Site Health
This pulls together:
- stale-date issues
- broken links
- image relevance alerts
- duplicate-image alerts
- metadata / OG / schema gaps
- recently failed or held QA checks

This is the site-health heartbeat panel.

---

## Module 3 — What’s On Control Panel
Shows:
- current active weekend window
- next weekend window
- current dispatch in use
- current homepage event anchors
- current What’s On event anchors
- drift alerts (expired event still referenced, wrong weekend window, etc.)

This matters because What’s On is the most visibly perishable surface.

---

## Module 4 — Content Hubs Status
Shows verticals/hubs such as:
- Dog-Friendly Peninsula
- Golf
- What’s On
- Places
- Eat
- Stay
- Wine

For each hub, show:
- number of live pages
- missing planned pages
- image completeness
- verification completeness
- metadata completeness
- notes / next action

Dog-Friendly Peninsula should be one of the first tracked hubs in Phase 1.

---

## Module 5 — Approvals Waiting
A lightweight panel showing:
- items waiting on James
- why they are held
- recommended action
- what is being given up by waiting

The goal is to reduce hidden blockers.

---

## Module 6 — Recent Deploys and Fixes
Shows:
- recent deploys
- recent content fixes
- recent cron actions
- recent live changes

This gives operational continuity without needing to search git, reports, and chat.

---

## 4. Data sources for Phase 1

Do not overengineer the data layer yet.
Use what already exists.

### Pull from:
- reports generated by cron jobs
- current hub/planned-page definitions from docs
- recent git/deploy activity
- QA issue files / known issue lists
- Mission Control docs/tasks where useful

### Phase 1 principle
**Assemble truth first, normalise later.**

---

## 5. Dog-Friendly Peninsula as a tracked hub

Phase 1 should explicitly track the Dog-Friendly Peninsula buildout.

### Suggested fields
- hub live? yes/no
- hub page exists? yes/no
- core four pages live? yes/no
- utility pages live? yes/no
- image quality state
- metadata completeness state
- verification completeness state
- internal linking completeness state
- next recommended action

This makes the Workbench immediately useful against current live work, not just future abstractions.

---

## 6. UI shape recommendation

### Layout
Use a dashboard-style card layout.

Recommended first screen:
- top row: Today’s Priorities / Approvals Waiting / Recent Deploys
- middle row: Live Site Health / What’s On Control Panel
- lower row: Content Hubs Status

### Tone
This should feel:
- editorial
- operational
- clear
- not enterprise-software ugly

### Important
The Workbench should feel like the command centre of a live publication, not a generic admin backend.

---

## 7. Phase 1 boundaries

### Include now
- visibility
- issue surfacing
- hub completeness
- state aggregation
- recommendations

### Do not include yet
- full inline editing
- heavy workflow automation UI
- complex approval management systems
- direct CMS replacement ambitions

This keeps the first build focused and shippable.

---

## 8. Implementation recommendation

### Best location
Build it inside or adjacent to Mission Control, rather than inventing a totally separate product surface.

Reason:
- faster to ship
- easier to reuse existing internal infrastructure
- keeps Peninsula operating surfaces together

---

## 9. Phase 1 success criteria

The first build is successful if James can open it and immediately know:
- what needs fixing
- what is blocked
- what’s live
- what’s broken
- how the dog hub is progressing
- whether What’s On is aligned with reality

If that works, the Workbench has earned Phase 2.

---

## 10. Next build after Phase 1

Once Phase 1 is working, Phase 2 should add:
- page-level readiness checklists
- richer hub completeness scoring
- stronger SEO/image/link drilldowns
- deeper issue ownership and routing

But not yet.

---

## 11. Final recommendation

Build the Phase 1 Workbench as a visibility layer first.

That is the correct first build.

It will immediately reduce:
- hidden drift
- operational confusion
- chat-only state management
- missed blockers

And it will make Peninsula Insider feel like a publication being run deliberately, not just edited reactively.
